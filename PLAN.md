# Splattar — Architecture & Roadmap

## Goal

A 100% Rust application that turns walkthrough video of an apartment into a high-definition Gaussian-surfel model you can explore first-person — desktop (WASD/mouse), VR (OpenXR), and a shareable web viewer (WASM/WebGPU). Dev machine: Windows 11 + NVIDIA GPU.

## Settled decisions

- **GPU via wgpu** (Vulkan/DX12/WebGPU compute) — full-speed on NVIDIA, no CUDA.
- **100% Rust, no C/C++ build deps** (no ffmpeg, COLMAP, OpenCV). System drivers/runtimes (GPU driver, OpenXR runtime DLL) are acceptable.
- **From scratch, as a learning project** — the differentiable rasterizer (WGSL forward + hand-derived backward), Adam optimizer, and densification are implemented here. Brush is studied as reference only.
- **Representation: 2D Gaussian surfels (2DGS)** with modern improvements — ray-splat-intersection rasterizer, depth-distortion + normal-consistency regularization, MCMC densification, mip-style anti-aliasing. Chosen over classic 3DGS because apartments are planar-dominant and surfels give view-consistent depth/normals → accurate surfaces, mesh export, collision geometry. SuGaR's goals (surface alignment + mesh) are achieved the 2DGS way: alignment is intrinsic to surfels; meshes come from TSDF fusion + marching cubes, not Poisson reconstruction (no pure-Rust support). Also surveyed and set aside: 3DGRT/3DGUT ray tracing (CUDA/OptiX-bound), SVRaster, feed-forward VGGT/VicaSplat (below per-scene-optimized fidelity), PGSR/GOF/RaDe-GS (PGSR-style losses kept as a stretch upgrade).
- **Video is the only product input — one or many videos.** No photo/snapshot mode. The pipeline exploits temporal structure (sequential frames, small baselines, a continuous camera path on a PTS timeline) instead of reproducing the legacy unordered-photo pipeline (global SfM + random-view batch training). Architecture is SLAM-shaped: tracking front-end → incremental submap mapping → Sim(3) graph alignment → short global refinement.
- **The scene is a graph of submaps.** A long video is segmented into overlapping sub-videos, each built independently (bounding drift); additional videos — e.g. a patch clip covering a missed room — register into the same graph via relocalization. Posed-frame-sequence dataset loaders exist only as internal validation harnesses, never as user-facing input.

**Highest-risk items, front-loaded in the roadmap:** (1) hand-derived WGSL backward pass; (2) pure-Rust visual-odometry pose pipeline; (3) pure-Rust video decode — hard-blocking, since video is the only input.

Design rules: the training rasterizer (`gs-kernels`) and viewing rasterizer (`gs-render`) are separate pipelines sharing `gs-wgpu` sort/infra; everything runs headless via `gs-cli` before any GUI exists; `gs-core` compiles everywhere including wasm32.

## Pipeline

1. **Streaming ingest:** MP4/.mov demux + decode with per-frame PTS; sharpness scoring; keyframe *promotion* (median flow displacement 15–40 px + sharpest-in-window) — non-keyframes still feed tracking.
2. **Tracking front-end (VO), two-pass:** the **causal pass** runs once in decode order — pyramidal KLT optical flow with constant-velocity motion prediction (small search windows; sequential frames make descriptor matching unnecessary in steady state), sharpness scoring, PTS bookkeeping, and a **zoom/lens-switch signal** (coherent radial flow no camera motion can explain = zoom; abrupt track-scale discontinuity = lens switch) — storing tracks per frame; **geometry runs anchor-out, not in decode order** (see §3/§5 — stored flow tracks are direction-agnostic data). Five-point + ARRSAC bootstrap at the anchor window; **continuous-time SE(3) trajectory spline parameterized by PTS** (smoothness prior fights drift on textureless walls, VFR-native, rolling-shutter-ready); sliding-window BA; **flow-inconsistency masks** exclude transients (people/pets) from supervision. Memory: full-res pixels kept only for promoted keyframes; tracking pyramids are discarded after the causal pass.
3. **Submap segmentation:** the stream is cut into overlapping submaps (~20–40 s, ~5 s overlap; boundaries preferentially at low-tracking-quality moments — blur, fast rotation). A detected zoom or lens switch **forces** a boundary: each submap owns an **intrinsics group** (focal, principal point, distortion) — new zoom level = new submap; continuous slow zoom splits at threshold crossings with a warning in the output. Within each segment, an **anchor window** is selected — the best-conditioned stretch (sharpness, texture density, adequate parallax), usually near the middle but chosen by quality, not position. Each submap owns its trajectory spline, surfel set, exposure state, and gauge (monocular scale is per-submap arbitrary); overlap keyframes belong to both neighbors. AKAZE descriptors of every promoted keyframe enter a project-wide relocalization database.
4. **Dense surfel initialization:** small-baseline plane-sweep depth on wgpu across neighboring frames → surfels born *on surfaces with normals from local plane fits* — skips most of densification's geometry discovery; the single biggest convergence lever.
5. **Incremental mapping (per submap, anchor-out):** the submap bootstraps at its anchor window, then grows *outward in both temporal directions*. Each promoted keyframe is first *tracked against the current splat model* (photometric alignment through the differentiable rasterizer — pose nearly free; both growth directions extend an established local map, so neither end gets a cold start); surfels spawn only in newly revealed regions; sliding-window optimization with **temporal minibatches** — adjacent keyframes see nearly the same splats, so tile bins/sorts are reused across steps (L1 + D-SSIM λ=0.2, depth-distortion + normal-consistency regularizers, mip-style 2D filter).
6. **Submap registration & graph alignment (deferred and continuous):** adjacent same-video submaps are constrained by shared overlap keyframes. Beyond that, *every* promoted keyframe — regardless of source video — is continuously matched against the project-wide relocalization DB (brute-force AKAZE; apartment scale needs no vocabulary): constraints attach **wherever** overlap occurs, at any time point in any video, never gated at a video's start. A match → matched 3D points → **Sim(3) estimate (7-DOF: per-submap scale)** → verified and polished by photometric alignment against rendered splats. All constraints feed a **Sim(3) pose graph over submaps**, relaxed via factrs/sparse LM — drift is distributed at overlaps, not baked into geometry; graph components merge the moment a constraint connects them. Overlap surfels are deduplicated during joint refinement (MCMC budget + opacity pruning keeps the better-supported copy).
7. **Global refinement (short):** all keyframes of all submaps; MCMC budget enforcement (2–4M surfels); progressive SH deg 0→2; joint pose-spline/focal refinement; **per-submap appearance splines** — a time-varying per-channel affine color transform (3× gain + offset) over PTS modeling auto-exposure *and* auto white balance together, regularized toward identity — plus cross-submap harmonization at graph merge. Appearance transforms are applied per-view in the loss and never baked into surfel SH colors; the scene stays canonical.
8. **Alignment & outputs:** per connected component — floor-plane RANSAC y-up + metric scale from a user reference measurement (default door ≈ 2.03 m; unconnected islands get provisional scale from a camera-height heuristic) → mesh extraction (median depth → TSDF fusion → marching cubes → simplified OBJ/GLB for collision + export, per island) → **archipelago assembly** when >1 component (presentation-level side-by-side layout on a shared floor) → **export = baked snapshot**: compose per-submap transforms, flatten to one world-space cloud, dedup, write .ply/.spz + a **scene-manifest JSON sidecar** (island bounds, spawn/teleport points, scale status) → viewers.

**Multi-session append:** a project directory persists keyframes, the descriptor database, the submap graph, the surfel model, and per-video calibration (a patch video may come from a different phone). `gs-cli run <video>` creates a project; `gs-cli add <video>` extends it. A new video always builds unconditionally as its own submap chain in its own gauge — a **floating island** — while registration runs continuously; islands merge the moment any overlap connects them (different starting points are irrelevant; only eventual overlap matters). **Unconnected islands are a normal in-progress state, not a failure:** the viewer and default export show *all* islands laid out side by side on a shared virtual floor with a clear gap in the horizontal (x/z) plane — the **archipelago view** — with status reporting ("2 islands, not yet connected"). This is itself the coverage diagnostic: the operator sees exactly which disjoint pieces exist and films a **bridge video** crossing between their territories; on merge, the layout collapses into true alignment. Capture guidance: ensure some part of each video covers ground another video covered; more shared coverage in more places = stronger alignment. Multiple videos ingest and track concurrently on CPU threads; the GPU builds submaps serially from a ready queue (see Scheduling decision).

Because mapping is incremental and submaps are independent by construction, a live preview can grow while the video plays; the target is a model minutes after ingest ends, not a multi-hour batch job.

## Key decisions

- **Primitive: 2DGS surfels, single training rasterizer.** Ray-splat intersection (exact perspective-correct evaluation of an oriented planar disk — no EWA affine approximation) yields per-pixel depth and normals for free. No parallel classic-3DGS training path — it would double the highest-risk work for little gain on planar indoor scenes. The viewer loads both surfel and standard 3DGS .ply layouts.
- **Surface quality: 2DGS regularizers; SuGaR by concept only.** Depth-distortion + normal-consistency losses, phased in after warm-up. Surfels are surface-aligned by construction, so SuGaR's SDF regularizer and Poisson meshing are unnecessary (and Poisson has no pure-Rust path). PGSR-style planar/multi-view consistency losses are the stretch upgrade if walls still ripple.
- **Mesh extraction: TSDF fusion + marching cubes** over rendered median depth from training views — simple, well-documented, pure-Rust-friendly. Output: apartment mesh, decimated for collision (walk mode: gravity + wall collision) and exportable OBJ/GLB.
- **Hand-rolled autodiff — no Burn.** The rasterizer backward is hand-derived WGSL regardless; the remaining chain rule (SH, activations, intersection Jacobian, pose grads) is bounded. Escape hatch: Burn glue in `gs-train` only, if derivations stall.
- **Pose: video visual odometry, not SfM.** KLT tracking + motion prediction, PTS-parameterized SE(3) spline, sliding-window BA; once mapping starts, new keyframes are posed photometrically against the model. Loop closure is not a separate subsystem — revisits register as submap constraints through the same relocalization machinery.
- **Scene = submap graph.** Per-submap gauge; Sim(3) registration (never assume shared scale between independently built monocular submaps); brute-force AKAZE relocalization; global pose-graph relaxation; merge-by-refinement with overlap dedup. One abstraction delivers bounded drift, multi-video/patch sessions, and pipeline overlap (submaps are independent units of work; CPU prepares the next while the GPU trains the current). Registration is deferred and continuous: new videos build as floating islands in their own gauge, constraints attach wherever overlap is found, and unconnected islands are first-class — shown as an archipelago (side-by-side on a shared floor) so the operator sees what to bridge. Island placement is a presentation-level transform per component, never part of the Sim(3) graph or geometry state; a bridge video's constraints replace layout with true alignment.
- **Anchor-out submap building.** Each submap bootstraps at its best-conditioned anchor window and grows bidirectionally, because: (1) VO drift is a random walk, so two ~N/2 chains halve worst-case drift at the segment ends *relative to the gauge anchor* — and the ends are exactly where Sim(3) seam constraints attach, so seam corrections shrink (end-to-end variance between the two ends is unchanged; resolving that is the graph's job); (2) segment boundaries sit at low-quality footage by construction, making them the worst possible place for fragile monocular bootstrap — anchor selection is the video analog of COLMAP's best-initial-pair; (3) both growth directions track against an established local map — no cold starts; (4) a tracking failure mid-half loses only that tail. Decode is forward-only (GOP structure), so the causal pass (decode → tracks → scores) is decoupled from geometry ordering — never conflate the two.
- **Incremental mapping over batch training.** Each keyframe starts from an already-good model; dense init skips discovery; redundant adjacent frames become a cache win (tile-bin/sort reuse) instead of wasted work. Drift is answered structurally by the submap graph, not by longer optimization.
- **Video decode (hard-blocking):** software decode behind a `VideoDecoder` trait. H.264: bake-off NihAV (`nihav-itu`) vs `rusty_h264` (bit-exact vs Cisco h264dec; Baseline+B-slices+most High), keep the winner. HEVC (iPhone default): validate `rust_h265` (pure Rust, Main/Main 10 4:2:0, new 2026-07) on real iPhone .mov streams. Vulkan Video NVDEC via `ash` is the hardware-decode stretch. A frame-sequence reader exists for internal test harnesses only. Beware parser-only crates masquerading as decoders (`media-codec-h265`, `scuffle-h26x`, `h264-reader`). iPhone specifics: .mov demux with hvcC/avcC length-prefixed → Annex B conversion; Dolby Vision decodes via the backward-compatible HLG base layer but needs HLG→SDR tone mapping; **VFR — always carry PTS, never frame_index/fps**.
- **Precision: f32 training, f16/quantized viewing.** WGSL has no f32 atomics — gradient accumulation uses per-tile shared memory flushed via u32-bitcast CAS. Viewer uses packed f16 (`shader-f16` where available) — bandwidth is the VR bottleneck.
- **One WGSL radix sort, two uses.** Training: (tileID ‖ depth) 64-bit keys → per-tile ranges. Viewer: 32-bit depth-only global sort per frame (web-splat-proven approach, >200 FPS class on NVIDIA).
- **SH degree 2 default** (deg 3 runtime option) — apartments are diffuse; saves ~40% splat payload for VR/web. Progressive unlock deg 0→1→2 during training.
- **Anti-aliasing:** 2DGS's object-space low-pass + screen-space 2D mip filter from the quality milestone — prevents thin-surfel shimmer in VR.
- **Appearance: time-varying affine color model. Intrinsics: per-submap groups with change detection.** Walking through a house sweeps auto-exposure and auto white balance continuously, so the camera has no fixed color parameters: each submap carries a per-channel affine color spline (gain + offset over PTS), applied in the loss only — never baked into surfels. Intrinsics are likewise not assumed constant per video: pinch-zoom, lens switches (0.5×/1×/3× modules = discrete focal jumps), focus breathing, and EIS crop/warp all occur in phone footage. Capture guidance says don't zoom, but the pipeline degrades gracefully — the causal pass detects zoom/lens switches and forces a submap boundary, so a new zoom level just becomes a new submap with its own jointly-refined intrinsics group; focus breathing is absorbed by per-submap focal refinement. Smartphone *local* (spatially varying) HDR tone mapping is the one appearance effect no global transform can undo — capture guidance + robust loss, explicitly not inverted in v1.
- **Scheduling: serial GPU training, pipelined CPU preparation.** Submap training saturates the GPU (bandwidth-bound rasterizer over millions of surfels); concurrent training jobs would time-slice the same bandwidth for no net throughput while ~doubling VRAM (params + grads + 2 Adam moments > 1 GB per active 2M-surfel instance) — and wgpu exposes one queue per device with no CUDA-streams-style async compute, so they'd interleave anyway. Instead, exploit the pipeline's heterogeneity: decode, KLT/sharpness, VO/BA, and relocalization are CPU work — run them concurrently across videos, staging ready submaps into a queue that the GPU drains serially (it never waits on ingest). Build order is correctness-irrelevant (the Sim(3) graph is order-independent), and serial training keeps runs reproducible and VRAM/MCMC budgeting simple. True parallel training = multi-GPU, out of scope.
- **Two-tier data model: appendable project, baked exports.** The project is the source of truth and the only appendable representation: submap surfels live in *submap-local coordinates forever*, with world placement composed at render/export time from the current Sim(3) solution (+ archipelago layout when disconnected) — never pre-baked into internal storage, so new constraints re-relax the graph for free. Exports (.ply/.spz/mesh + scene-manifest sidecar) are lossy baked snapshots: transforms composed, cloud flattened, overlaps deduped, training state dropped. **Never re-ingest an export** — submap structure, gauges, and the descriptor DB are gone; `add <video>` grows the project, then re-export. The manifest sidecar (island bounds, spawn/teleport points, scale status) lets our viewers keep walk-mode confinement and island teleport even from a baked file; external viewers ignore it.
- **Splat file formats.** No standardized surfel interchange exists, so `gs-io` writes three layouts: (1) **native surfel .ply** — 2 scales (s_u, s_v) + rotation, our lossless documented layout; (2) **compat 3DGS .ply** — zero third scale appended (the 2DGS reference convention), opens in every standard viewer; (3) **.spz** (Niantic, ~10× smaller than ply, quantized) — inherently 3-scale, surfels stored flattened the same way. SPZ versioning under the purity constraint: write gzip-era SPZ (v2/v3) via flate2's default pure-Rust backend now; SPZ 4 (ZSTD parallel streams, no 10M cap, SH deg 4, vendor extensions) becomes writable only when a pure-Rust ZSTD *encoder* matures — reading SPZ 4 via `ruzstd` is unblocked sooner. SPZ 4 vendor extensions are a candidate in-file carrier for the scene manifest; the JSON sidecar stays primary. Stretch: glTF export via KHR_gaussian_splatting + spz compression extension (Khronos RC, the standards-track interop path).

## Milestones (ordered by risk; each ends in a runnable demo)

- **M0 — Render an existing splat file.** Workspace scaffold; gs-core/io/wgpu/render/viewer; `gs-cli view`. Accept: `samples/ply/cactus-high.ply` (~1.9M splats, SH deg 3) at ≥120 FPS @1440p; side-by-side visually matches a reference viewer (`cactus-low` becomes the golden-test fixture).
- **M1 — GPU primitives hardened.** Radix sort, prefix sum, tile binning as standalone tested kernels. Accept: property tests vs CPU pass (random sizes 1–10M, order + stability); 4M-key sort < 2 ms (timestamp queries).
- **M2 — Differentiable 2DGS rasterizer (risk #1).** CPU analytic oracle first; then WGSL ray-splat forward; then hand-derived backward through intersection, opacity, SH, and pose/focal; depth + normal render targets. Accept: GPU↔CPU-analytic ≤1e-4 rel; analytic↔finite-diff ≤1e-2 rel per parameter class on randomized micro-scenes; 50-surfel image overfit reaches PSNR > 35 dB.
- **M3 — Trainer validated on posed video sequences** (internal harness; isolates trainer from pose errors). Adam-in-WGSL, SSIM kernels, posed-sequence loader (Replica renders / ScanNet-style). Accept: PSNR within ~1 dB of published 2DGS on comparable indoor scenes; 30k iters ≤ 60 min.
- **M4 — Quality & geometry.** MCMC + mip filters + progressive SH + depth-distortion & normal-consistency regularizers. Accept: ≥ M3 PSNR at a fixed 2M budget; no zoom shimmer; fewer floaters; clean normal maps on flat walls (visual golden + normal-vs-depth-gradient metric); Chamfer vs Replica GT mesh in the published 2DGS ballpark.
- **M5 — Video ingest** (gates everything downstream). Streaming MP4/.mov demux with PTS (VFR-safe); H.264 bake-off; `rust_h265` iPhone validation (10-bit + HLG tone mapping) against reference decodes of real clips; color-correct YUV→RGB (golden frames); sharpness scoring.
- **M6 — VO front-end (risk #2; parallelizable after M2).** Causal pass (KLT tracking + motion prediction, sharpness/PTS bookkeeping, zoom/lens-switch detection), five-point bootstrap, PTS-parameterized SE(3) spline, sliding-window BA, transient masks, keyframe promotion, submap boundary policy incl. intrinsics-change boundaries, **anchor-window selection + anchor-out estimation**, keyframe descriptor DB + relocalization primitive (AKAZE → Sim(3) → photometric verify). Validate on GT-trajectory video (TUM RGB-D mono / ScanNet / Replica renders). Accept: ATE < 1% of trajectory; RPE < 0.5°; training on our poses loses < 1 dB vs GT poses and photometric refinement recovers ≥ half; relocalization ≥ 90% on held-out overlapping clips, zero false positives on non-overlapping ones; A/B on one GT sequence logged showing anchor-out beats forward-only on seam-adjacent drift; a test clip with a mid-video lens switch yields a forced submap boundary and a converging reconstruction (no smeared geometry).
- **M7 — Video-native training, end-to-end on a real apartment video.** Plane-sweep dense depth → surface-aligned init; incremental mapping (track-against-model, spawn-in-new-regions, temporal minibatches with sort/bin reuse) built as submaps from day one, anchor-out; Sim(3) graph relaxation + overlap dedup; global refinement; per-submap appearance splines (per-channel affine over PTS). `gs-cli run walkthrough.mp4` → walk your own apartment. Accept: held-out-keyframe PSNR > 24 dB; no double-wall seams in overlap regions (visual golden + normal-map check); total pipeline ≤ 60 min for a 2-min 1080p video (stretch: model ready minutes after ingest ends); logged ablation showing dense-init + incremental beats naive batch on time-to-PSNR.
- **M8 — Multi-video & patch sessions.** Project persistence; `gs-cli add <video>`; deferred continuous cross-video Sim(3) registration; island handling + bridge merging; per-submap color harmonization. Accept: two videos with *different starting points* and only mid-video overlap merge coherently with no ghosting at seams; a patch clip covering a missed area merges; two videos with no mutual overlap render as separated islands on a shared floor in viewer and export, with clear connectivity status; a third bridge video overlapping both merges them into one component and the presentation offset disappears (seamless single scene); after an export, `add` continues from the project and a re-export reflects the merge (exports are snapshots, never re-ingested).
- **M9 — Mesh extraction & collision.** Median depth → TSDF fusion → marching cubes → decimation, per island; `gs-cli export --mesh` (OBJ/GLB); viewer walk mode (gravity + capsule-vs-mesh collision, toggleable vs fly; walk is confined per island — fly or teleport spans the archipelago). Accept: no holes bigger than a doorway on main surfaces; walk mode can't pass through walls/floor; export opens in Blender.
- **M10 — Web viewer.** wasm32 build, progressive .spz streaming + scene-manifest support (island spawn/teleport). Accept: ≥60 FPS @1080p in Chrome WebGPU with a 1.5M-splat apartment; shareable via any static host.
- **M11 — VR.** OpenXR + Vulkan interop, stereo with shared cyclopean-eye sort, f16 packing, LOD, resolution scaling; scale-calibration UX; M9 collision for roomscale comfort bounds. Accept: sustained 90 Hz on the target headset; world scale within ±5% after calibration.
- **M12 — Stretch.** Live-growing preview during ingest (decode/track/map concurrently); deeper submap pipeline overlap (CPU prep concurrent with GPU training, occupancy-bubble filling — true parallel training is multi-GPU, out of scope); near-real-time reconstruction; Vulkan Video NVDEC (HEVC/iPhone); egui pipeline UI; foveation; PGSR-style consistency losses; glTF KHR_gaussian_splatting export; SPZ 4 write support (pure-Rust ZSTD encoder dependent).

## Risks

| Risk | Mitigation |
|---|---|
| Silent wrong gradients / atomics races in WGSL backward | CPU oracle + FD checks in CI gate all kernel work; shared-mem accumulation; test Vulkan and DX12 backends |
| Decode is hard-blocking; `rust_h265` is v0.1 and unproven on real phone streams; Dolby Vision/HLG color | decoder validation front-loaded to M5 with golden-frame checks; H.264 capture guidance; internal frame-sequence harness keeps trainer/VO work unblocked; Vulkan Video in M12 |
| Pose drift on textureless walls / auto-exposure | short submap chains bound accumulation; spline smoothness prior; capture guidance; appearance splines; photometric track-against-model refinement; held-out PSNR canary |
| Incremental mapping bakes early errors into the model (vs batch's global view) | submap graph is the structural answer — drift confined per submap, resolved by Sim(3) relaxation at overlaps; sliding-window BA before surfels spawn; global refinement; M7 ablation vs batch harness |
| Submap seam misalignment → double walls / ghosting | photometric Sim(3) polish after descriptor registration; overlap dedup during joint refinement; seam golden tests in M7/M8 acceptance |
| Continuous auto-exposure/AWB drift within a video, and cross-session lighting shifts (different time of day or phone) | per-submap time-varying affine appearance splines + cross-submap harmonization at merge; per-submap intrinsics groups; capture guidance; patch seams checked in M8 |
| Smartphone local (spatially varying) HDR tone mapping — unrecoverable by any global per-frame transform | capture guidance (lock exposure / non-HDR video mode where possible); robust loss tolerance; explicitly not inverted in v1 |
| EIS crop/warp + focus breathing = small continuous intrinsics wobble | absorbed by per-submap focal refinement + spline smoothness; not explicitly modeled v1; logged if residuals show it |
| A video never registers (insufficient overlap with the rest of the project) | not a failure mode: unconnected islands render as an archipelago (the coverage diagnostic itself) and are bridged by a later video; inter-island scale is provisional until connected (camera-height heuristic); ≥90% relocalization gate with no-false-positive requirement in M6 |
| wgpu-compute perf gap vs CUDA (no f32 atomics, historical warp-op gaps) | subgroup ops on native Vulkan; shared-mem accumulation; accept 2–3× (M3's 60-min budget assumes it); profile via timestamp queries from M1 |
| VR frame budget (2 × 90 Hz at headset res) | f16 packing, SH deg 2, shared per-frame sort, LOD, resolution scale, foveation stretch |
| 2DGS fidelity dip on fine/fuzzy detail (plants, fabric) vs 3DGS | accept ~0.5 dB; apartments are planar-dominant; MCMC budget headroom; PGSR losses or SH deg 3 per scene if needed |
| Monocular scale/orientation wrong for VR | floor-plane RANSAC for gravity; user reference measurement; ±5% acceptance in M11 |
| Rolling shutter + motion blur in handheld footage | sharpness-based keyframe promotion; capture guidance; spline is rolling-shutter-ready but explicit modeling is out of scope v1 |
| Solo-dev scope creep | every milestone ends in a runnable `gs-cli` demo; one shared render crate serves all three viewers; escape hatches documented but not taken by default |

## Verification

- **Gradient three-way agreement** (finite-diff ↔ CPU analytic ↔ GPU WGSL) on randomized micro-scenes per parameter class — in CI forever; any kernel edit reruns them.
- **Golden-image tests** from M0: deterministic camera paths over fixture scenes (`cactus-low`), PSNR ≥ 45 dB tolerance (not byte hashes — drivers differ).
- **Property tests** (M1): sort order/stability, binning coverage vs CPU.
- **Dataset benchmarks** on posed video sequences — Replica renders (PSNR/SSIM + Chamfer vs GT mesh), TUM RGB-D / ScanNet (ATE/RPE) — tracked in a checked-in results log. These loaders are internal harnesses; the product input remains video only.
- **Relocalization benchmark** (M6+): ≥90% success on held-out overlapping clips; non-overlapping clips rejected (no false positives).
- **Seam golden tests** (M7+): deterministic camera paths across submap boundaries; overlap regions stay free of double walls (visual golden + cross-seam normal-map agreement).
- **Time-to-PSNR ablations** (M7+): incremental + dense-init vs naive batch on the same scene — speed claims are measured, not assumed.
- **End-to-end canary** (M7+): one fixed real apartment video; held-out PSNR + wall-clock tracked per change.
- **Perf gates** via the `gs-wgpu` timestamp-query bench harness: sort < 2 ms @ 4M keys; viewer forward < 4 ms @ 1080p/2M splats; train iteration < 100 ms.
- **Failure-source isolation:** trainer validated on GT poses (M3) before our poses (M6→M7); ingest validated on golden frames (M5) before feeding the pipeline. Never debug pose and trainer against the same failing run.
